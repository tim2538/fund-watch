# Deploy to GitHub Pages (github.io/fund-watch)

This project is configured to build a **static export** (`output: "export"`, folder `out/`)
and deploy it to GitHub Pages via GitHub Actions — the workflow lives at
`.github/workflows/deploy.yml`.

Result URL: `https://<your-username>.github.io/fund-watch/`

> Because GitHub Pages only serves static files, there is no live server. The
> Finnomena data is fetched **at build time** and baked into the HTML. The
> workflow re-runs **daily** (and on every push) to refresh it. Change the
> `schedule:` cron in the workflow to refresh more/less often.

---

## One-time setup

### 1. Name the repo `fund-watch`
The site is served from a sub-path (`/fund-watch`), so the repo name must match the
`NEXT_PUBLIC_BASE_PATH` in the workflow.

- If your repo has a different name, edit `.github/workflows/deploy.yml` and set
  `NEXT_PUBLIC_BASE_PATH: /<your-repo-name>`.
- (Special case) If you deploy to a **user site** repo named `<user>.github.io`,
  the app lives at the domain root — set `NEXT_PUBLIC_BASE_PATH:` to empty (`""`).

### 2. Generate and commit the lockfile
The workflow uses `npm ci`, which requires `package-lock.json`:

```bash
npm install          # creates package-lock.json (also lets you run/test locally)
git add package-lock.json
```

### 3. Push the project to GitHub

```bash
git init
git add .
git commit -m "Fund Watch PWA"
git branch -M main
git remote add origin https://github.com/<user>/fund-watch.git
git push -u origin main
```

### 4. Turn on GitHub Pages (Actions source)
On GitHub: **Settings → Pages → Build and deployment → Source = "GitHub Actions"**.
(You do *not* pick a branch — the workflow publishes the artifact.)

That's it. The push triggers the workflow; watch progress under the **Actions** tab.
When it finishes, the deploy step prints the live URL.

---

## Everyday use

- **Update the site:** push to `main` → it rebuilds and redeploys automatically.
- **Refresh data now (no code change):** Actions tab → "Deploy to GitHub Pages" →
  **Run workflow** (manual trigger), or wait for the daily scheduled run.

---

## Test the static build locally (optional)

```bash
# Build exactly like Pages does (with the sub-path)
NEXT_PUBLIC_BASE_PATH=/fund-watch npm run build

# Serve ./out so the /fund-watch prefix resolves
npx serve out -l 3000
# then open http://localhost:3000/fund-watch/
```

For normal local development (no sub-path), just run `npm run dev` and open
`http://localhost:3000` — `NEXT_PUBLIC_BASE_PATH` is empty in dev.

## Notes / troubleshooting

- **PWA / service worker:** generated by `next-pwa` at build and enabled in
  production only. The manifest (`src/app/manifest.ts`) and icons include the
  basePath so "Add to Home Screen" works under `/fund-watch/`. If the service
  worker 404s after a rename, hard-refresh once to clear the old registration.
- **No API routes.** The dashboard fetches Finnomena and embeds all fund data
  into the page at build time, so the app works fully offline. (Route handlers
  were removed because a `/api/funds` file collides with the `/api/funds/*`
  folder during static export — `EISDIR`.)
- **Custom domain?** Add a `public/CNAME` file with your domain and set
  `NEXT_PUBLIC_BASE_PATH:` to empty in the workflow.
- Data shown reflects the last build/refresh time, not real-time NAV.
